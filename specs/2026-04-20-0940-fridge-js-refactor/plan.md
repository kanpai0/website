# fridge.js — refactor for testability and readability

## What was built

The inline `<script>` block in `layouts/index.html` (~90 lines) was split into:

- `static/js/fridge.js` — all filter logic and DOM wiring
- An inline `<script>` block in `index.html` that only defines `FRIDGE` and `FLAVORS`

`fridge.js` was then progressively refactored into a structured, testable file.

## Final structure

```
// — Pure logic       parseDataList, hasRequiredIngredients, matchesActiveFlavors, isCardVisible
// — DOM queries      getActiveIngredients, getActiveFlavors
// — DOM updates      updateCardVisibility, updateFilterSummary, applyFilters
// — Persistence      loadFridge, saveFridge, loadFlavors, saveFlavors
// — Init             init()
```

## Why

The original script had no separation of concerns: filtering algorithm, DOM mutations, localStorage access, and event listener wiring were interleaved. This made it impossible to unit-test any part of the logic without a full browser context.

## Key decisions

**Extract pure functions first.** The filtering algorithm (`isCardVisible`) was the highest-value extraction — it takes plain arrays and returns a boolean, requiring no DOM. It was further split into `hasRequiredIngredients` and `matchesActiveFlavors` so each concern is individually testable.

**`parseDataList` in Pure logic, not DOM updates.** Parsing `data-fridge` / `data-flavors` strings into arrays is a string operation with no DOM dependency. Belongs with the other pure functions.

**`FRIDGE` stays dynamically generated by Hugo, `FLAVORS` stays manual.** `FRIDGE` is derived at build time from all recipe frontmatter (`fridge[]` arrays), ensuring it stays in sync with content automatically. `FLAVORS` is hardcoded in the template to preserve display order — a data file would add complexity for no gain.

**`fridgeCb/fridgePanel/fridgeBtn` queried once inside `init()`, captured by closure.** These are permanent DOM elements, so querying once at init is correct. Moving the query inside `syncFridgeAria` would re-query on every toggle unnecessarily. File-scope globals were rejected in favour of closure capture inside `init()` for tighter scoping.

**localStorage reads moved to Persistence as `loadFridge`/`loadFlavors`.** The reads were previously inlined in `init()`. Naming them symmetrically with `saveFridge`/`saveFlavors` makes the persistence boundary explicit and keeps `init()` focused on wiring.

**Event listener wiring stays in `init()`.** Only the localStorage reads belong in Persistence — attaching listeners is an init concern, not a persistence concern.

## Unit tests unlocked

`parseDataList`, `hasRequiredIngredients`, `matchesActiveFlavors`, and `isCardVisible` can all be tested with plain Node.js (no DOM, no browser):

- empty `fridgeNeeded` → always visible
- all ingredients present → visible
- one ingredient missing → hidden
- no active flavor filter → always visible
- card has all active flavors → visible
- card missing one active flavor → hidden
- combined AND logic across both axes

## Pre-commit validation added

A new check in `scripts/pre-commit.sh` verifies that every `flavors[]` slug used in recipe frontmatter is listed in the `FLAVORS` constant in `layouts/index.html`. Mirrors the existing fridge schema check.
